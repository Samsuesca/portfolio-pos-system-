2. Los invariantes canónicos que faltan

Todo el plan gira en torno a hacer ciertos estados imposibles. Tres invariantes, hoy no enforzados:

I1 — Reserva canónica (por fila de inventario, llaveada por product.school_id, NULL=global): inventory.reserved_quantity == Σ order_items.quantity_reserved WHERE reserved_from_stock = true AND item_status IN ('pending','in_production','ready') # "abiertos" I2 — Estado derivado (salvo CANCELLED y override auditado): order.status == derive_order_status(items_activos) # all delivered → DELIVERED ; algún delivered + resto abierto → PARTIALLY_DELIVERED ; # all ready/delivered → READY ; algún in_production → IN_PRODUCTION ; else PENDING I3 — Item terminal sin reserva: item_status IN ('delivered','cancelled') ⟹ quantity_reserved = 0 AND reserved_from_stock = false

Hoy se violan los tres: I1 derivó (ya reconciliado), I2 lo violan 31 encargos, I3 lo violan 146 items. El plan los enforça en runtime (servicio) y, donde es seguro, en DB (CHECK de I3) + un checker periódico (I1/I2).

Los arquitectos propusieron caminos en conflicto. La síntesis adversarial eligió (no fusionó), porque dos dueños del mismo dato es la receta del bug original:

5. La secuencia segura (6 fases)

El orden importa: ciertas piezas son prerequisitos duros de otras. Desplegar fail-closed sin notificaciones-after-commit sería activamente peligroso.

FASE 0Prerequisitos — WS0 (resolver cadena alembic) → WS1 (decisión de arquitectura) → confirmar empíricamente el casing de los enums (order_status UPPERCASE, item_status lowercase) antes de escribir cualquier SQL.

FASE 1Solo código, reversible — WS2 (notificaciones after-commit) → WS3 (generadores contra available + matar modelo viejo) → WS4 (reset alcanzable). Desplegar y observar: dejan de generarse stale y drift nuevos.

FASE 2fail-closed — WS5, solo tras WS2 desplegado. Path por path, consumir-antes-de-notificar.

FASE 3Data-fix (ventana, conteo físico CERRADO) — WS7. Idempotente, toca solo reserved_quantity y campos de item, nunca quantity ni orders.status.

FASE 4Estados (ortogonal a reservas) — WS10 (ADD VALUE) → WS6 (derive + PATCH) → WS8 (re-sync 24+7) → WS12 frontend en el mismo PR que WS6/WS8.

FASE 5Constraints + defensa en profundidad — WS11 (CHECK I3, tras 0 violaciones) → WS9 (checker en lifespan, AUTO_REMEDIATE=False).

Regla crítica de no-colisión

La remediación de estados (WS8: los 24+7) y la del modelo de reservas (WS7: los 146 stale) son ortogonales — commits y migraciones distintos. Mezclarlas arriesga que un UPDATE de stock toque orders.status por accidente. Y nada de FASE 3/4/5 se corre con el conteo físico abierto (colisiona adjust_quantity vs reserved≤quantity).

6. La regla transversal: notification-safety

Cambiar orders.status dispara email/WhatsApp al cliente. Dos consecuencias de diseño que el plan respeta:

  • WS2 es prerequisito de TODO fail-closed: hoy las notificaciones salen dentro de la transacción (update_status 67-113). Si un fail-closed hace rollback después de notificar, el cliente recibe "tu pedido está listo" de un estado que no se guardó. Por eso las notificaciones se mueven a after-commit primero.
  • La remediación de los 24+7 (WS8) es 100% SQL crudo: un UPDATE orders SET status=… no ejecuta el código Python que notifica. Solo se permite bajar de READY (nunca subir), con dry-run y cross-check contra order_audit_overrides. El sistema no des-notifica a los 24 clientes que ya recibieron "listo" por error — esa decisión (mensaje correctivo) es del owner, caso a caso.

7. Remediar los 24+7 encargos incoherentes (la UX de las vendedoras)

El objetivo que pediste — "que los encargos tengan sentido" — se cierra en WS6+WS8+WS12:

  • WS8 (datos): bajar los 24 READY-falsos a su estado real (17→PENDING, 7→IN_PRODUCTION) y los 7 atascados a PARTIALLY_DELIVERED. SQL crudo, notification-safe, solo bajar.
  • WS6 (código): que de ahora en adelante order.status se derive de los items — imposible volver a tener un READY falso. Nuevo estado PARTIALLY_DELIVERED para entregas parciales (hoy se disfrazan de READY).
  • WS12 (UI): la card del producto muestra "Reservado: N (a M encargos)" con el encargo que sostiene cada reserva, y un botón ver/liberar — se acaba la percepción de "fantasma" sobre reservas legítimas (el caso del layaway de Geraldine).

Decisión del owner pendiente: los 24 clientes ya recibieron "tu pedido está listo" por error. El re-sync corrige el sistema pero no des-notifica. ¿Mensaje correctivo a esos clientes, o se deja pasar? El sistema no decide esto.

8. Bloqueos y prerequisitos (leer antes de arrancar)

  • WS0 — cadena alembic sin commitear: hay 5 migraciones B2B/branches + modelos sin commitear en el working tree. Las migraciones de este plan (WS10, WS11) deben encadenar sobre un head real y commiteado. Decisión del owner: ¿se commitea la cadena B2B antes, o se desacopla? El diseño no puede usar down_revision='branch_retrofit_001' mientras ese parent no esté commiteado.
  • Casing de enums: order_status guarda NAMES en MAYÚSCULA, item_status en minúscula (values_callable). Todo SQL de migración debe confirmarlo con SELECT DISTINCT primero — mezclarlos da el peor fallo: 0 filas matcheadas, falso éxito.
  • WS10 es irreversible-en-caliente: ALTER TYPE … ADD VALUE no se revierte sin downtime. Es el punto de no retorno del sprint de estados.
  • Ventana de despliegue: FASE 3/4/5 requieren el conteo físico CERRADO.

Apéndice — invariantes del checker (WS9)

# I1 — reserva canónica (CRÍTICO, solo alerta, nunca auto-remedia)
SELECT count(*) FROM inventory i
LEFT JOIN (SELECT oi.product_id, p.school_id, SUM(oi.quantity_reserved) backing
  FROM order_items oi JOIN products p ON p.id=oi.product_id
  WHERE oi.reserved_from_stock AND oi.item_status IN ('pending','in_production','ready')
  GROUP BY 1,2) b ON b.product_id=i.product_id
   AND (b.school_id=i.school_id OR (b.school_id IS NULL AND i.school_id IS NULL))
WHERE i.reserved_quantity <> COALESCE(b.backing,0);          # debe ser 0

# I3 — item terminal con reserva colgada (auto-remediable: idempotente, no toca inventory)
SELECT count(*) FROM order_items
WHERE item_status IN ('delivered','cancelled') AND quantity_reserved > 0;   # debe ser 0
