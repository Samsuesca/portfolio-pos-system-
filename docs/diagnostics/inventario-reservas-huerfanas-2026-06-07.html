Reservas de inventario inconsistentes: por qué no se puede ajustar stock (y por qué hay riesgo de sobreventa oculto)

Las vendedoras reportan errores consistentes al ajustar inventario durante el conteo físico: el stock existe pero el sistema lo rechaza con Insufficient inventory. Available: 0. La investigación reveló tres defectos de datos/código, no uno — y un cuarto bug de percepción en la UI que hizo el síntoma confuso.

Fecha: 7 jun 2026 Sistema: UCR — uniformesconsuelorios.com (v3.1.0) Severidad: Alta — operativa + integridad Estado: Diagnóstico cerrado · fix de desbloqueo validado, pendiente aplicar

Documento master consolidado: fusiona dos investigaciones independientes (sesión A: síntoma + catch-22 + script de reconciliación clamp-down; sesión B: verificación adversarial + cuantificación de la sub-reserva + conexión con la conciliación de encargos + causa del bug de UI). Resumen ejecutivo

El error no es "no hay stock". El stock físico existe, pero el campo reserved_quantity de cada producto quedó desincronizado de las reservas reales de encargos abiertos. La causa madre es una migración de backfill (inv_reserved_qty, 2-may-2026) con dos defectos, más un tercer defecto de código vivo en el consumo de reservas. El resultado son dos caras del mismo problema:

  • Sobre-reserva (+8 uds, 7 productos): reservas fantasma que bloquean el ajuste de inventario. Esto es lo que ven las vendedoras.
  • Sub-reserva (−9 uds, 6 productos globales): reservas legítimas que se perdieron del inventario → riesgo de sobreventa (oculto hasta esta investigación).
  • Fuga viva: el flujo de entrega traga errores y puede seguir generando huérfanos hoy.

1. El síntoma

Al hacer el conteo físico, la vendedora abre "Ajustar Inventario", elige − Remover y guarda. El sistema rechaza la operación en productos específicos:

Insufficient inventory. Available: 0 (quantity=1, reserved=1), Requested: 1
Insufficient inventory. Available: 0 (quantity=2, reserved=2), Requested: 1

El recuadro muestra "Stock actual: 1" — la prenda existe, pero el sistema no la deja tocar. El mensaje estaba en inglés (contra la convención del proyecto y opaco para la operadora).

En los logs de producción se ve la sesión de conteo (7 jun, 20:03–20:26): decenas de POST /inventory/product/{id}/adjust con 200, y productos puntuales repitiendo 400 (uno con 9 fallos seguidos: 0248d09d…). El conteo avanza, salvo en los productos atascados — de ahí la sensación de "problema consistente".

2. La mecánica del bloqueo

El primitivo de ajuste (InventoryService.adjust_quantity) usa un UPDATE atómico con esta guarda:

# backend/app/services/inventory.py:240
.where(Inventory.quantity + delta >= Inventory.reserved_quantity)

Para remover 1 unidad de un producto con quantity=1, reserved=1:

1 + (−1) >= 1   →   0 >= 1   →   FALSO   →   0 filas actualizadas   →   error 400

El available = quantity − reserved = 0. La guarda es correcta: no debes poder remover stock comprometido a un pedido. El problema no es la guarda — es que reserved_quantity está mal.

✓ Ya corregido (working tree): el mensaje de error del ajuste ahora es en español y explica el porqué: "Stock insuficiente. Disponible: 0 (total=1, reservado=1)… Hay 1 unidad(es) reservada(s) a pedidos; libere la reserva para poder ajustar." — services/inventory.py.

3. Los tres defectos y su causa común

reserved_quantity debería cumplir, por producto, este invariante:

inventory.reserved_quantity == Σ order_items.quantity_reserved WHERE reserved_from_stock AND item_status NOT IN (delivered, cancelled)

No lo cumple. La columna se creó con la migración inv_reserved_qty (2-may-2026) y su backfill tiene dos bugs; un tercer bug vive en el flujo de consumo.

Defecto 1 — Sobre-reserva (orphans): el backfill filtró por estado del PEDIDO, no del ITEM

# alembic/versions/inv_reserved_qty_add_reserved_quantity.py:41-44
WHERE oi.reserved_from_stock = true
  AND oi.quantity_reserved > 0
  AND o.status IN ('PENDING','IN_PRODUCTION','READY')   # ← estado del PEDIDO
  # ← falta:  AND oi.item_status NOT IN ('delivered','cancelled')

Un encargo parcialmente entregado queda en status=READY aunque algunos de sus items ya estén item_status=delivered (los enums de pedido e item son independientes). El backfill, al filtrar por el estado del pedido, sumó la reserva de esos items ya entregados a inventory.reserved_quantity. Esas son las reservas fantasma.

CONF-CHO-Ama-10 ← CONFAMA-001-ENC-2026-0018 order_status = READY · item_status = delivered · reserva = 1 (fantasma)

Timeline que lo prueba: los 9 items phantom se finalizaron entre el 26-ene y el 5-feb, todos antes del backfill (2-may). El backfill contó items entregados meses atrás. En el modelo viejo, quantity_reserved nunca se limpiaba al entregar, así que casi todo item finalizado pre-backfill arrastra ese valor.

Defecto 2 — Sub-reserva: el backfill no tocó los productos globales

# misma migración, líneas 46-55
... GROUP BY oi.product_id, o.school_id            # ← colegio del PEDIDO
UPDATE inventory inv ... WHERE inv.product_id = pr.product_id
  AND ( (inv.school_id IS NULL AND pr.school_id IS NULL)  # global
        OR inv.school_id = pr.school_id );

Para un producto global (GLB-…) la fila de inventario tiene school_id = NULL, pero el pedido tiene un school_id concreto. Entonces la rama NULL/NULL falla (pr.school_id no es NULL) y la rama de igualdad falla (NULL = UUID es UNKNOWN en SQL). El backfill nunca actualizó la fila global. Resultado: 6 productos globales quedaron con reserved = 0 aunque tienen encargos abiertos que reservan 1–3 unidades cada uno. El sistema cree que ese stock está disponible cuando en realidad está comprometido → sobreventa.

Defecto 3 — Fuga viva: el consumo de reserva traga la excepción

# backend/app/services/order/status.py:331-348  (_consume_item_reserved_stock)
try:
    await inv_service.consume_reserved_stock(...)   # exige reserved >= amount
    item.quantity_reserved = 0                       # ← solo corre si NO hubo excepción
    await self.db.flush()
except Exception as e:
    logger.error(...)                                # ← traga el error, NO re-lanza, NO resetea

Si el consumo atómico falla (porque inventory.reserved ya estaba en 0, p. ej. por el Defecto 2), la línea item.quantity_reserved = 0 queda inalcanzable: el item se marca delivered con la reserva colgada. El mismo patrón fail-open existe en cancellation.py:73 y changes.py:217. Evidencia viva: el item CONF-CAM-Azu-04 del encargo CONFAMA-001-ENC-2026-0040 se entregó el 4-may (post-backfill) y conserva quantity_reserved=1 sin liberar. Hoy es inofensivo (la fila de inventario ya estaba en 0), pero prueba que el generador sigue activo.

5. El misterio de la UI: "está reservado pero no aparece en encargos"

Observación del dueño: en la fila del producto se ve el badge reservado > 0, pero al mirar los encargos no aparece ningún pedido reservando esas unidades. Esto hace que toda reserva parezca "fantasma", aunque algunas sean legítimas. Hay una causa de código precisa:

# backend/app/api/routes/products.py:205-208
# READY/DELIVERED = already fulfilled, should NOT count
pending_item_statuses = [OrderItemStatus.PENDING, OrderItemStatus.IN_PRODUCTION]

El contador de "encargos pendientes" por producto excluye los items READY (los considera "ya cumplidos"). Pero la reserva de stock persiste durante READY — solo se libera al pasar a delivered/cancelled. Hay un desajuste de semántica: para la UI un encargo READY "no está pendiente", pero para el inventario sigue reservando.

Caso real: CONF-SUD-Ama-10 (el de tu captura)

Sus 2 unidades reservadas son el par de layaway de Geraldine Ramírez:

CONFAMA-001-ENC-2026-0007 order=READY · item=ready · reserva=1 → LEGÍTIMA (apartado real) CONFAMA-001-ENC-2026-0018 order=READY · item=delivered · reserva=1 → FANTASMA (entregado, no liberó)

La legítima (ENC-0007) no aparece como "pendiente" porque su item está ready (excluido del contador). La fantasma (ENC-0018) está entregada, así que tampoco figura como reservando. Por eso ves reservado=2, pendientes=0 y parece todo fantasma — pero solo 1 de las 2 lo es. Ambos encargos son de Geraldine y ambos están en el acta de conciliación (casos 24 y 25).

Bugs de visualización hermanos (erosionan la confianza en los números): los productos globales también muestran pending_orders=0 por el mismatch de school_id, y el método alterno get_products_with_inventory arma su propio pending_count con la misma exclusión. El vendedor ve reservado>0, pendientes=0 y asume "fantasma" aunque sea legítimo.

6. La conexión con la conciliación de encargos (los 25)

El dueño sospechaba que esto se relacionaba con la conciliación de encargos del acta (GATE 0, v3.1.0). Es correcto, pero con un matiz importante:

  • La causa de las reservas inconsistentes es el backfill (Defectos 1 y 2) + el fail-open (Defecto 3). No lo causó la auditoría contable de encargos.
  • Pero los productos afectados son precisamente los de esos encargos. La sobre-reserva viene de ENC-0018, ENC-0068 y ENC-0096; la sub-reserva la causan JUCUM (ENC-0131), Santi Mazo (ENC-0058) y Jennifer (ENC-0096); CONF-SUD-Ama-10 es el layaway de Geraldine (ENC-0007/0018). Seis de los 25 encargos del acta están enredados en esto.
  • La auditoría del acta arregló la plata (CxC, caja, P&L) pero, por diseño, no tocó el estado operativo (orders.status, item_status, inventario). Por eso "no fue suficiente para estabilizar los encargos": faltaba la dimensión de inventario/reservas de esos mismos pedidos parcialmente entregados.

En otras palabras: los encargos huérfanos del acta y las reservas huérfanas del inventario son dos síntomas del mismo desorden operativo — pedidos que se entregaron físicamente pero el sistema nunca marcó como entregados a nivel de pedido. El acta cerró la cara contable; este diagnóstico cierra la cara de inventario.

7. El catch-22

La válvula de escape está bloqueada por lo que debe arreglar

El propio código documenta la remediación de una reserva colgada como "usar el endpoint de adjust" — pero eso es imposible para estos productos: adjust exige quantity + delta ≥ reserved, y con reserved = quantity no se puede remover nada. No existe ningún endpoint que reduzca reserved_quantity sin un pedido detrás (release_stock/consume viven solo dentro del ciclo de vida del encargo). La vendedora queda sin salida desde la interfaz. Por eso se siente "consistente": son exactamente los productos que cayeron en este estado.

9. La solución y la decisión pendiente

9.1 Desbloqueo inmediato (Cara A) — script validado

Script idempotente canónico backend/scripts/reconcile_reserved_quantity.py. Fórmula por fila:

target = min(reserved_quantity_actual, respaldo_de_encargos_abiertos)
se aplica SOLO si  target < reserved_quantity_actual     # clamp-DOWN: nunca sube

Guardas (cada una cierra un ataque adversarial verificado): aborta ante product_id NULL abierto · agrupa por product.school_id con partición NULL explícita · filtro item_status en minúsculas · UPDATE optimista (sin row lock) · log de auditoría con quantity_delta = 0 (la reserva no mueve stock físico; meterlo ahí contaminaría COGS).

venv/bin/python -m scripts.reconcile_reserved_quantity            # dry-run (rollback)
venv/bin/python -m scripts.reconcile_reserved_quantity --commit   # persiste

Libera las 8 uds fantasma y desbloquea el conteo HOY. No toca quantity físico (el conteo en curso es la fuente de verdad) ni ninguna reserva legítima.

Nota de consolidación: existió un segundo script (reconcile_inventory_reservations.py, SET absoluto que subía y bajaba) creado en una investigación paralela. Se eliminó: subir reserved automáticamente contradice una decisión de diseño auditada y es riesgoso. La Cara B se trata aparte y con decisión explícita (abajo).

9.2 La sub-reserva (Cara B) — requiere tu decisión

Los 6 globales sub-reservados son reservas legítimas perdidas (encargos abiertos reales: JUCUM, Santi, Jennifer…). Dos opciones:

  • Recomendada — corregir (subir reserved al respaldo real): compromete esas 9 uds a sus encargos. Reduce el disponible vendible pero elimina el riesgo de sobreventa. Es seguro (0 filas violan la constraint). Se haría con una corrección dirigida y explícita (no el script clamp-down), porque para JUCUM (6 uds de blusas) la sobreventa sí importa.
  • Alternativa — dejar como está: mantiene el disponible alto pero un cliente o venta de mostrador podría llevarse stock prometido a JUCUM. Solo aceptable si confías en que el stock se repone a tiempo.

9.3 Hardening de código (para que no recurra)

El fix de datos es un parche sobre código aún roto. Sin esto, la deriva se reacumula. Ver §11.

10. Verificación adversarial

El diagnóstico se sometió a dos revisiones multi-agente independientes (≈10 y 11 agentes; ~0.9M y ~1.1M tokens) que atacaron las hipótesis buscando refutarlas y corromper el script de escritura.

✓ Las tres hipótesis: CONFIRMADAS (alta confianza)

Mecánica del rechazo, backfill por order.status (over), backfill por order.school_id (under) y fail-open en consume — todas verificadas con file:line y consistentes con el timeline de prod. El bug está VIVO, no solo legado.

⚠ Ataques que cambiaron el plan

  • FK product_id ON DELETE SET NULL: si un producto borrado dejó un item con reserva viva y product_id NULL, un INNER JOIN a products lo descartaría → el script vería respaldo 0 → liberaría una reserva legítima. → El script aborta si detecta esto (hoy: 0 casos).
  • Auto-subir reserved (Cara B): el plan paralelo de SET absoluto habría bloqueado ventas legítimas y, en filas con poco stock, violado reserved ≤ quantity. → Descartado; Cara B se decide aparte.
  • Enum case-sensitivity: OrderItemStatus guarda minúsculas, OrderStatus MAYÚSCULAS — mezclarlos colapsaría el respaldo y liberaría todas las reservas. → Filtro en minúsculas, verificado.
