🔍 Auditoría Exhaustiva de API REST — Uniformes System API v2.0.0
Especificación: OpenAPI 3.1 | Total de rutas: 363 | Total de operaciones: 477 | Schemas: 399 | Tags: 45

1. Diseño de URIs y Naming Conventions — 7/10
Aspectos positivos: El versionado /api/v1/ es correcto y consistente en el 99% de las rutas. Se utiliza kebab-case uniformemente en los segmentos de URI (como balance-accounts, garment-types, change-password), sin mezcla de camelCase ni snake_case en los paths. Los nombres de recursos usan plural de forma consistente (schools, users, products, clients, sales, orders). Los path parameters usan snake_case ({school_id}, {user_id}), lo cual es consistente.
Problemas encontrados:
La inconsistencia más grave son 3 rutas que usan /api/v1/global-garment-types/ y /api/v1/global-products/ con guion, mientras que las otras 154 rutas globales usan /api/v1/global/garment-types/ con sub-ruta. Esto rompe la uniformidad del patrón de namespace. Los endpoints de Health (/ping, /health) viven fuera del prefijo /api/v1/, lo cual es aceptable para probes de infraestructura pero podría ser más limpio. Se detectan endpoints de acción como /activate, /pay, /cancel, /approve que, aunque son un patrón aceptable para operaciones RPC-like que no encajan en CRUD puro, su proliferación (más de 25 action endpoints) sugiere que se podrían modelar mejor como transiciones de estado vía PATCH al recurso principal.

2. Estructura de Recursos y Verbos HTTP — 8/10
Aspectos positivos: La distribución de verbos es saludable: 234 GET (49%), 133 POST (28%), 30 PUT (6%), 42 PATCH (9%), 38 DELETE (8%). No hay verbos en las URIs (no se detectó ningún segmento como /getUsers o /createSale). No hay GET ni DELETE con request body, lo cual respeta las convenciones HTTP. Los POST para creación retornan 201 de forma consistente (48 de 49 endpoints de creación). Las acciones no-CRUD como /pay, /approve, /cancel usan POST correctamente.
Problemas encontrados:
Hay inconsistencia entre PUT y PATCH: algunos recursos usan PUT para update completo (como schools, users, products con 30 endpoints) y otros usan PATCH para update parcial (clients, expenses, employees con 42 endpoints), pero no existe un patrón claro que determine cuándo se usa uno u otro. La mezcla sugiere decisiones ad-hoc más que una convención definida. Idealmente, PATCH debería usarse para actualizaciones parciales y PUT para reemplazos completos, y la API debería ofrecer ambos o estandarizar uno. Además, POST /api/v1/global/accounting/expenses/check-balance y POST /api/v1/cash-drawer/validate-access tienen semánticas de consulta (son idempotentes y de lectura), lo que sugiere que deberían ser GET con query parameters.

3. Agrupación y Organización — 6/10
Aspectos positivos: Los 45 tags agrupan los endpoints por dominio de negocio de forma que un desarrollador puede encontrar lo que busca. La separación entre Accounting (school-scoped) y Global Accounting es lógica para el modelo multi-tenant. El patrón de separación entre portal de clientes (Client Portal, Order Portal) y el back-office es coherente.
Problemas encontrados:
Con 45 tags, la API sufre de fragmentación excesiva. Tags como Workforce - Shifts, Workforce - Attendance, Workforce - Checklists, Workforce - Performance, Workforce - Responsibilities podrían consolidarse bajo un único tag Workforce. Lo mismo aplica para Reports y Global Reports, Accounting y Global Accounting, Products y Global Products. Hay un tag documents en minúsculas mientras todos los demás usan Title Case, lo cual es una inconsistencia de naming. Existen 18 recursos top-level (/api/v1/products, /api/v1/clients, /api/v1/sales, etc.) que coexisten en paralelo con los mismos recursos school-scoped (/api/v1/schools/{school_id}/products), creando ambigüedad sobre cuál usar. Además, recursos como cash-drawer, business-info, telegram-alerts y cfo-dashboard están al nivel raíz cuando probablemente deberían estar bajo un namespace como /global/ o /admin/. Los tags no tienen descripciones documentadas en la spec (spec.tags está vacío), lo que dificulta la orientación del desarrollador.

4. Autenticación y Seguridad — 7.5/10
Aspectos positivos: Se usa HTTP Bearer Token (JWT), un esquema estándar y robusto. De 477 operaciones, 437 (91.6%) están protegidas con security requirement. Los endpoints no protegidos son razonables: login, registro de portal, health checks, y endpoints públicos de consulta de escuelas. Se documentan consistentemente los códigos 401 (445 endpoints) y 403 (443 endpoints), indicando que hay control de autenticación y autorización.
Problemas encontrados:
POST /api/v1/portal/orders — un endpoint de creación de pedidos — está sin autenticación. Crear órdenes sin autenticación es un riesgo significativo de abuso (spam, DDoS a la base de datos, fraude). Esto requiere revisión urgente. Varios GET de productos y garment-types globales están abiertos (GET /api/v1/global/garment-types/{id}, GET /api/v1/global/products/search, etc.); aunque podrían ser intencionales para el portal público, deberían evaluarse. No hay mención de esquemas de seguridad complementarios (rate limiting, API keys para clientes externos, OAuth scopes, CORS). El login menciona rate limiting en la descripción (5 intentos por minuto por IP), pero esto no está formalizado en la spec via extensiones OpenAPI.

5. Consistencia y Convenciones — 6.5/10
Aspectos positivos: Los operationId siguen camelCase de forma 100% consistente en las 477 operaciones (createSchool, listUsers, getMe, etc.). Los path parameters usan snake_case uniformemente. Las propiedades de los schemas también usan snake_case (school_id, client_id, created_at, updated_at). El content-type es consistentemente application/json.
Problemas encontrados:
La inconsistencia PUT vs PATCH ya mencionada es el problema de convención más serio. Hay inconsistencia en códigos de respuesta para DELETE: 8 endpoints retornan 200, mientras 30 retornan 204, sin un criterio claro (por ejemplo, DELETE /api/v1/users/{user_id} retorna 200 pero DELETE /api/v1/schools/{school_id} retorna 204). El tag documents en minúsculas rompe la convención Title Case. Las 3 rutas con patrón /global-resource vs /global/resource ya mencionadas. No existe un esquema de error estandarizado propio; se depende del HTTPValidationError de FastAPI (422) pero no hay un modelo de error de negocio unificado para 400/404/409.

6. Completitud CRUD — 8/10
Aspectos positivos: Los recursos principales tienen operaciones CRUD completas: Schools (POST, GET list, GET by ID, PUT, DELETE + acciones), Users (CRUD completo + gestión de roles), Products (CRUD + inventario), Clients (CRUD + students sub-resource), Sales y Orders (CRUD + flujos de estado completos). Las sub-recursos como schools/{id}/products, schools/{id}/sales están bien modelados. Los flujos de negocio complejos como pagos, alteraciones, y contabilidad tienen endpoints específicos.
Problemas encontrados:
Algunos recursos tienen CREATE pero no DELETE (como transactions de contabilidad, lo cual podría ser intencional por audit trail). Faltan endpoints de búsqueda o filtrado avanzado para muchos recursos que probablemente lo necesiten (solo 13 endpoints tienen parámetro search de 186 endpoints de listado). No hay endpoint de batch/bulk DELETE. Hay endpoints potencialmente redundantes: los recursos top-level /api/v1/products y /api/v1/garment-types que listan desde todas las escuelas duplican funcionalidad que podría resolverse con un filtro en los endpoints de escuela.

7. Paginación, Filtrado y Ordenamiento — 4/10
Aspectos positivos: Existe un modelo genérico PaginatedResponse bien diseñado con campos items, total, skip, limit, page, total_pages y has_more, lo cual es una buena práctica. Se usan query params skip y limit como parámetros de paginación.
Problemas encontrados:
Este es uno de los puntos más débiles de la API. De 63 endpoints con semántica de listado (identificados por el summary "List..."), solo 10 (15.9%) retornan una respuesta paginada; los otros 53 retornan arrays planos sin paginación. Esto es un riesgo operacional serio: si alguna tabla crece significativamente, las respuestas sin paginar podrían causar timeouts, consumo excesivo de memoria, y degradación del servicio. Solo 1 endpoint tiene soporte para ordenamiento (sort). Solo 13 endpoints soportan búsqueda textual. No hay convención de filtrado estandarizada; los parámetros varían enormemente entre endpoints (active_only, with_inventory, status, date_from, date_to aparecen inconsistentemente). No hay soporte para fields selection (sparse fieldsets) ni para cursor-based pagination (más escalable que offset-based).

8. Códigos de Respuesta HTTP — 7.5/10
Aspectos positivos: Buena variedad de códigos documentados: 200 (382), 201 (65), 204 (30), 400 (125), 401 (445), 403 (443), 404 (218), 409 (14), 422 (426). El uso de 201 para creaciones, 204 para eliminaciones, 409 para conflictos es apropiado. La cobertura de errores de autorización (401/403) es prácticamente universal.
Problemas encontrados:
No se documenta el código 500 en ningún endpoint. Aunque los errores de servidor son inesperados, documentar una respuesta genérica 500 con un schema de error es una best practice para que los consumidores sepan qué esperar. La inconsistencia 200 vs 204 en DELETE ya mencionada (8 vs 30). El código 400 solo se documenta en 125 de 477 endpoints; muchos endpoints de mutación carecen de documentación de errores de validación de negocio (distinct de 422 de validación de schema). El 404 se documenta en 218 endpoints, pero algunos endpoints con path parameters no lo documentan.

9. Schemas y Modelos de Datos — 7/10
Aspectos positivos: 399 schemas es una cobertura robusta. Se sigue la convención de DTOs separados: *Create (53), *Response (133), *Update (45), *ListResponse (34). Esto es una excelente práctica que evita exponer modelos de base de datos directamente. 307 schemas (77%) tienen campos required definidos. Se usan 40 enums tipados (SaleStatus, PaymentMethod, AccountType, etc.), lo que mejora la validación. Se usan referencias ($ref) extensivamente, promoviendo la reutilización.
Problemas encontrados:
De 2890 propiedades totales en los schemas, solo 68 (2.4%) tienen descripción a nivel de campo. Esto es extremadamente bajo y dificulta la comprensión de qué significa cada campo sin contexto adicional. No hay un solo ejemplo (example) en toda la especificación, lo que obliga al desarrollador a adivinar formatos, valores válidos, y payloads de prueba. No hay un schema de error de negocio estandarizado; solo existe HTTPValidationError de FastAPI. Falta un ErrorResponse genérico con campos como code, message, details. Algunos schemas como SaleCreate tienen todos los campos opcionales excepto items, lo cual podría permitir creaciones incompletas.

10. Documentación y Developer Experience (DX) — 5.5/10
Aspectos positivos: 459 de 477 operaciones (96.2%) tienen descripción. Todas las 477 operaciones tienen operationId único y en camelCase, facilitando la generación de SDKs. Las descripciones de los endpoints incluyen detalles útiles como la mención de rate limiting en el login. La API usa OAS 3.1, la versión más reciente de la especificación.
Problemas encontrados:
La información de la API (info) es mínima: no hay contact, license, termsOfService, ni externalDocs. No hay tags con descripciones (el array tags en la raíz de la spec está vacío), dejando al desarrollador sin contexto sobre qué hace cada grupo. 18 endpoints carecen completamente de descripción (incluyendo todos los de Google OAuth). No hay servers definidos, lo que dificulta probar la API sin configuración manual. La ausencia total de examples en los schemas es crítica para el DX; un desarrollador no puede construir un request sin leer código fuente o hacer reverse engineering. No hay guía de autenticación (cómo obtener un token, cómo refrescarlo, cuánto dura). No hay documentación de errores (qué estructura tienen, qué significan los códigos). Los 2822 campos sin descripción significan que el desarrollador tiene que adivinar el significado de campos como is_historical, source, student_grade, etc.

🔴 Top 5 Problemas Críticos
1. Paginación ausente en el 84% de los endpoints de listado. 53 de 63 endpoints que retornan listas devuelven arrays planos sin paginación. Esto es una bomba de tiempo: a medida que la base de datos crezca, estos endpoints causarán timeouts, consumo excesivo de memoria y potencialmente caídas del servicio. Es el problema más urgente de resolver.
2. Creación de órdenes sin autenticación (POST /api/v1/portal/orders). Permitir la creación de recursos transaccionales sin ningún tipo de autenticación expone la API a abuso masivo, inyección de datos basura, y potencial DDoS a la base de datos. Necesita al menos autenticación de portal o un mecanismo anti-spam.
3. Ausencia total de examples en 399 schemas y 2822 campos sin descripción. Un desarrollador que integre esta API tendrá que hacer trial-and-error para construir requests válidos. Esto multiplica exponencialmente el tiempo de integración y los errores. Ningún estándar de la industria (Google API Guidelines, Microsoft REST Guidelines) aceptaría esto en producción.
4. Inconsistencia PUT vs PATCH sin convención clara. 30 endpoints usan PUT y 42 usan PATCH para actualización, sin un criterio documentado de cuándo usar cada uno. Esto confunde a los consumidores y genera bugs de integración cuando un dev asume un comportamiento y encuentra otro.
5. Fragmentación de namespace y rutas inconsistentes. 3 rutas usan /global-resource mientras 154 usan /global/resource. Los recursos existen tanto en top-level como en school-scoped sin documentación de cuándo usar cada uno. El tag documents en minúsculas. Todo esto evidencia falta de una guía de estilo API enforced.

🟢 Top 5 Fortalezas
1. Modelo de DTOs bien estructurado. La separación en *Create, *Response, *Update, *ListResponse para cada recurso es una práctica de arquitectura madura que protege contra over-posting, permite evolución independiente de inputs y outputs, y facilita la generación de código tipado.
2. Cobertura de seguridad robusta. El 91.6% de endpoints están protegidos con Bearer token. La documentación consistente de 401 y 403 en prácticamente todos los endpoints protegidos demuestra que hay una capa de autorización seria implementada (no solo autenticación).
3. Modelo de PaginatedResponse genérico bien diseñado. Cuando se usa, el modelo de paginación con items, total, skip, limit, page, total_pages, has_more es completo y ergonómico. El problema es que no se usa en suficientes endpoints, pero la base está bien establecida.
4. OperationId consistente al 100%. Las 477 operaciones tienen operationId únicos en camelCase, lo que permite generación limpia de SDKs en cualquier lenguaje sin conflictos ni ambigüedades.
5. Cobertura funcional completa del dominio. La API cubre un dominio de negocio complejo (ventas, órdenes, contabilidad, inventario, RRHH, nómina, roles, portal de clientes, alertas Telegram, modelo financiero, cola de impresión) con 477 operaciones bien organizadas. La completitud CRUD es alta y los flujos de negocio complejos están modelados con endpoints de acción específicos.

🚦 Veredicto: ¿Lista para producción?
NO en su estado actual. La API tiene una base arquitectónica sólida (buenos DTOs, seguridad aceptable, cobertura funcional), pero presenta riesgos operacionales críticos que deben resolverse antes de un go-live con carga real. La ausencia de paginación en la mayoría de los endpoints de listado es un show-stopper que causará problemas de rendimiento en cuanto los datos crezcan. La falta de examples y descripciones a nivel de campo hará que la integración por terceros (o nuevos miembros del equipo) sea dolorosamente lenta.
Si esta API ya está en producción con bajo volumen de datos y consumidores internos, puede funcionar temporalmente, pero necesita las mejoras del roadmap antes de escalar.

🗺️ Roadmap de Mejoras Priorizadas
🔥 Quick Wins (1-2 sprints, alto impacto)
QW1 — Agregar paginación a los 53 endpoints de listado sin paginar. Ya existe el modelo PaginatedResponse genérico; aplicarlo a los endpoints faltantes es principalmente mecánico. Priorizar: sales, orders, products, clients, inventory, transactions.
QW2 — Proteger POST /api/v1/portal/orders. Agregar al menos autenticación de portal o un mecanismo de validación (CAPTCHA, token temporal).
QW3 — Corregir las 3 rutas inconsistentes /api/v1/global-garment-types/ y /api/v1/global-products/ para alinearlas con /api/v1/global/.
QW4 — Estandarizar DELETE a 204 No Content en los 8 endpoints que actualmente retornan 200.
QW5 — Capitalizar el tag documents a Documents.
🔧 Mejoras Estructurales (3-5 sprints)
ME1 — Definir y documentar convención PUT vs PATCH. Migrar todos los updates a PATCH (parcial) y reservar PUT solo cuando se requiera reemplazo completo. Documentar la decisión en una API Style Guide interna.
ME2 — Agregar examples a todos los schemas. Como mínimo, agregar un ejemplo completo por cada *Create y *Response schema (los 186 más utilizados). Esto transforma la experiencia del desarrollador.
ME3 — Crear un ErrorResponse estandarizado con campos error_code, message, details, timestamp, y documentarlo como respuesta para 400/404/409/500 en todos los endpoints.
ME4 — Agregar descripciones a tags y enricher info. Documentar contact, license, servers, termsOfService. Agregar descripción a cada tag. Agregar descriptions a los campos de schemas más relevantes (al menos los 50 schemas más usados).
ME5 — Consolidar tags de 45 a ~25-30, agrupando los módulos Workforce y las duplicaciones School/Global.
🏗️ Cambios Arquitectónicos (6+ sprints)
CA1 — Estandarizar filtrado y ordenamiento. Definir convención global para query params (sort_by, order, filter[field]), implementar en todos los endpoints de listado.
CA2 — Resolver la dualidad top-level vs school-scoped. Documentar explícitamente cuándo usar /api/v1/products vs /api/v1/schools/{id}/products, o eliminar una de las dos vías. Considerar hacer que los top-level sean admin-only con filtro ?school_id=.
CA3 — Evaluar cursor-based pagination para los endpoints de alto volumen (sales, orders, transactions, inventory logs) donde offset-based pagination degrada con datasets grandes.
CA4 — Implementar API versioning strategy documentada para cuando inevitablemente se necesiten breaking changes (el /v1/ está, pero no hay política de deprecación ni headers de versión).
